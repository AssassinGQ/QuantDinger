<objective>
Extend the Phase 1 sufficiency contract and logging layer so Phase 2 execution code can (a) represent orchestration failures as a typed insufficient outcome, (b) emit a stable `ibkr_open_blocked_insufficient_data` audit event with an explicit payload schema, and (c) keep operator-visible error text bounded and non-sensitive — without yet wiring `SignalExecutor` (that is Plan `02-02`).
Output: updated types, logging helpers, and tests; no changes to `signal_executor.py` in this plan.
</objective>

<review_carryover source="02-REVIEWS.md / 02-CONTEXT.md R-01–R-09">
- **R-01:** PLAN and executor treat **02-CONTEXT** as authoritative for execution-path exception mapping; Phase 1 service docstring remains library-level “propagate” — Phase 2 façade (Plan 02-02) catches.
- **R-02 / N3:** Phase 2 MUST NOT emit `ibkr_insufficient_data_alert_sent` (Phase 3). Emitters in this plan MUST NOT reference that event name.
- **R-03:** Stable `ibkr_open_blocked_insufficient_data` payload keys listed in Task 2 `<action>` (audit contract for Phase 3 correlation).
- **R-04:** Summary max **200** chars; category `schedule`|`kline`|`unknown`; no secrets in structured payloads.
- **R-05:** Intent/`target_weight` classification is Plan 02-02; this plan only supplies types + blocked payload builder consumed there.
- **R-06–R-09:** Joint gate, reduce negative test, kline alignment, batch perf — verification lives in Plan 02-02; types/logging must not block those tests.
</review_carryover>
